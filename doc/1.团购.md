# 团购（拼团）功能文档

## 一、功能概述

团购功能允许用户发起拼团活动，邀请好友共同购买套餐，达到成团人数后可享受团购价。

---

#### 核心业务逻辑

**开团 (`start`):**
- 检查套餐存在、用户无进行中的团购
- 默认 `minCount = 2`
- 过期时间 = 当前 + 48小时
- 开团人自动成为第一个参团者（JOINED）

**参团 (`join`):**
- FAILED 活动不可参团，SUCCESS 活动可参团
- 检查是否已参团（防重复）
- 参团后检查人数是否达到 `minCount` → 自动成团

**成团 (`completeGroupBuy` — 私有方法):**
- 更新活动状态 → SUCCESS
- 更新所有参团者状态 → SUCCESS
- 发放 9 折优惠券给每位参团者（GROUP_BUY 类型，30天有效，限该套餐）

**过期处理：**
- 定时任务每小时执行一次（`group-buy.scheduler.ts`）
- 将已过期的 ACTIVED 活动设为 FAILED

### 2.3 订单关联

**wx-order 创建流程（`wx-order.service.ts` 第 188-207 行）：**

订单创建后自动关联团购活动：
1. 查询用户是否参与了该套餐的 ACTIVE/SUCCESS 团购活动
2. 如果找到匹配的活动，将 `groupBuyActivityId` 设置到订单上
3. 此操作为非阻塞（失败不影响下单）

**`orders.service.ts` 的列表和详情查询**均包含 `groupBuyActivity`（id/status/minCount）用于管理后台显示。

### 2.4 优惠券

- 类型：`GROUP_BUY`（`couponType: 'GROUP_BUY'`）
- 折扣：`PERCENT` 类型，`discountValue: 10`（9折）
- 适用范围：`applicableType: 'PACKAGE'`，限该套餐
- 有效期：30天
- 防重复：检查 `UserCoupon` 是否已存在

#### 页面结构

| 页面 | 路径 | 说明 |
|------|------|------|
| 开团页 | `pages/group-buy/start/start` | 选择成团人数，开团 |
| 团购详情 | `pages/group-buy/detail/detail` | 状态、倒计时、参团、海报生成、分享 |

#### 核心流程

```
套系详情 → 点击"拼团优惠" → 开团页 → 开团 → 团购详情页
                                                ↓
                     参团 ← 扫码/分享 ← 团购详情页 ← 其他人
                      ↓
              分享到朋友圈/保存海报（ShareGate）
                      ↓
               POST /group-buy/:id/join
                      ↓
         预约流程（日期 → 时段 → 信息 → 确认）
                      ↓
                wx-order/create（创建订单，自动关联团购活动ID）
                      ↓
                /pages/payment/payment（支付）
```

**分享机制：**
- 参团前必须分享到朋友圈或保存海报（ShareGate）
- 支持分享给好友（`onShareAppMessage`）
- 支持分享到朋友圈（`onShareTimeline`）
- 标题格式："快来参团！{套餐名称} 仅需¥{价格}"

**海报生成：**
- Canvas 2D API 绘制（375×750px）
- 包含：海报标题、描述、主图、套餐名称、价格、参与者头像、二维码
- 二维码类型：微信小程序码 + 标准二维码双模式

## 三、当前已知问题

### 3.1 团购价未应用于订单

**现状：** 团购价仅用于前端展示，订单创建时使用套餐标准价。`wx-order.service.ts` 只关联 `groupBuyActivityId` 到订单，不修改订单金额。

**影响：** 参团用户支付的不是团购价，而是标准价。团购失去了核心价值。

**涉及代码：**
- `wx-order.service.ts` `createOrderFromCart()` — 创建订单后关联活动，不修改 totalAmount
- `orders.service.ts` `create()` — 无团购相关逻辑

### 3.2 Booking 流程下单未传递团购价

**现状：** 参团后进入预约流程，预约确认页面计算 `totalAmount` 时未使用 `groupPrice`。

**影响：** 即使后端修复了价格应用，前端传递给后端的金额可能仍然是标准价。

### 3.3 退款未更新参团状态

**现状：** 团购订单退款后 `GroupBuyParticipant.status` 仍为 SUCCESS，退款后 9 折券仍然可用。

**涉及代码：**
- `GroupBuyParticipant` 定义了 `REFUNDED` 状态但没有任何代码使用
- 退款流程（`payments.service.ts`）不区分是否来自团购

### 3.4 `orders.service.ts` 创建订单未关联团购

**现状：** 仅 `wx-order.service.ts` 的创建流程有团购关联逻辑。后台创建的订单不会关联团购活动。

### 3.5 GroupBuyList 与 OrderGroupBuyList 重复

**现状：** `Marketing/GroupBuyList.tsx`（功能完整的 CRUD 页面）和 `Orders/OrderGroupBuyList.tsx`（带统计的团购订单页）是两个独立的团购管理页面。前者有创建功能但未被路由引用，后者在 `/marketing/group-buy` 路由下但缺少创建功能。

## 五、团购价格在订单/退款中的关系

```
套餐设置
└── groupPrice: 团购价（仅展示）
    └── 小程序详情页显示团购价
        └── 参团 → 预约流程 → 创建订单
            ├── wx-order/create（自动关联 groupBuyActivityId）
            │   └── totalAmount = 标准价 ❌（应为 groupPrice）
            └── 退款
                ├── GroupBuyParticipant.status 仍为 SUCCESS ❌
                └── 优惠券仍可用 ❌
```
