对账管理模块功能分析

  模块定位
  对账管理是系统的资金安全防线，负责自动检测订单与支付记录之间的金额差异和状态不一致，确保系统账目与第三方支付平台数据一致。与支付管理模块深度联动，作为兜底机制防止资金损失。

  ---
  对账机制总览（双轨制）

  ┌─────────────────────┬────────────────────────┬──────────────────────────────┐
  │      对账类型       │       执行频率         │            职责              │
  ├─────────────────────┼────────────────────────┼──────────────────────────────┤
  │ 每日全量差异检测     │ 每日凌晨 3:00 (CST)   │ 检测订单金额、支付状态不一致  │
  ├─────────────────────┼────────────────────────┼──────────────────────────────┤
  │ 微信支付状态同步     │ 每 5 分钟              │ 主动查询微信侧支付状态，兜底  │
  └─────────────────────┴────────────────────────┴──────────────────────────────┘

  ---
  后端定时任务（2个）

  1. 每日对账任务 (reconciliation.task.ts)
     - 每日凌晨 3:00 自动执行
     - 三种差异检测类型：

     类型1: 金额不匹配
     ┌─────────────────────────────────────────────────────────┐
     │ 订单 paymentStatus = FULLY_PAID                        │
     │ 但 paidAmount ≠ totalAmount（容忍 0.01 元误差）         │
     │ 严重级别：差额 > 100 = 高风险                         │
     │           差额 10~100 = 中风险                        │
     │           差额 0.01~10 = 低风险                       │
     └─────────────────────────────────────────────────────────┘

     类型2: 状态不一致
     ┌─────────────────────────────────────────────────────────┐
     │ 订单 paymentStatus = PARTIAL_PAID                      │
     │ 但实际 paidAmount >= totalAmount                       │
     │ 严重级别：高风险                                       │
     └─────────────────────────────────────────────────────────┘

     类型3: 支付状态不匹配
     ┌─────────────────────────────────────────────────────────┐
     │ 订单 paymentStatus = PENDING_PAYMENT                    │
     │ 但存在状态为 PAID 的支付记录                            │
     │ 严重级别：高风险                                       │
     └─────────────────────────────────────────────────────────┘

     - 发现差异后：写入日志 + 发送管理员通知
     - 支持手动触发：POST /payments/reconciliation/manual

  2. 支付状态对账任务 (payment-reconciliation.scheduler.ts)
     - 每 5 分钟执行一次
     - 查找创建超过 5 分钟、状态为 PENDING_PAYMENT 的订单
     - 调用微信支付 V3 查询订单真实状态
     - 如微信侧显示 SUCCESS 则自动同步更新
     - 作为微信回调丢失的兜底机制

  ---
  前端页面（5个路由）

  1. 对账管理页面 (ReconciliationManagement.tsx)
     路由：/reconciliation/management

     功能：
     - 显示自动对账时间信息（每日凌晨3:00）
     - 手动执行对账按钮（调用后端手动对账接口）
     - 差异检测规则说明（3种类型 + 严重级别判定）
     - 对账历史时间线（模拟数据，展示执行记录）
     - 上次执行时间显示

  2. 对账中心首页 (Reconciliation.tsx)
     路由：/reconciliation

     功能：
     - 5个标签页切换：

     ┌────────────────────────────────────────────────────────────┐
     │ 标签页            │ 内容                                  │
     ├──────────────────┼────────────────────────────────────────┤
     │ 对账概览          │ 统计卡片 + 检索条件 + 对账明细表格     │
     │ 订单金额统计     │ OrderAmountStatistics 子组件           │
     │ 退款统计         │ RefundStatistics 子组件                │
     │ 收款统计         │ PaymentStatistics 子组件               │
     │ 凭证统计         │ 开发预留                              │
     └────────────────────────────────────────────────────────────┘

     - 统计卡片：对账天数、匹配天数、总差异金额、总交易金额
     - 检索条件：日期范围、支付平台（全部/微信/支付宝/现金）
     - 对账明细表格：日期、平台、订单数量、系统金额、平台金额、差异金额、状态
     - 差异金额颜色标识：绿色（匹配）/ 红色（不匹配）
     - 状态标签：已匹配（绿色）/ 有差异（红色）/ 待处理（蓝色）
     - 行操作：查看详情、标记异常
     - 合计行：汇总所有数据
     - 导出功能：Excel（xlsx 格式） + PDF（jspdf 格式）
     - 更多操作：导出全部交易明细、重新对账、对账设置
     - 冲账退款弹窗：RefundCorrectionModal

  3. 对账简版页面 (ReconciliationSimple.tsx)
     - 比对账中心功能简化
     - 仅对账概览标签页
     - CSV 格式导出
     - 预留冲账退款标签页（开发中）

  4. 增强对账看板 (EnhancedReconciliation.tsx)
     路由：/reconciliation（与 Reconciliation.tsx 同路由）

     功能：
     - 顶部告警横幅
     - 今日/本月统计卡片
     - 待处理告警列表（预警信息 + 数量）
     - 可疑支付标签页（重复支付、超额支付、系统错误）
     - 对账详情表格

  5. 可疑支付检测 (SuspiciousPayments.tsx)
     功能：
     - 风险统计卡片（总计/高风险/中风险/低风险）
     - 按严重级别筛选
     - 按类型标签页（全部/重复支付/超额支付/系统错误）
     - 数据表格：支付单号、订单号、客户、金额、类型、严重级别、原因、检测时间
     - 标记已处理（含备注）

  ---
  子组件（4个）

  1. OrderAmountStatistics.tsx — 订单金额统计视图
  2. RefundStatistics.tsx — 退款统计视图
  3. PaymentStatistics.tsx — 收款统计视图
  4. RefundCorrectionModal.tsx — 冲账退款弹窗

  ---
  后端 API（对账相关，2个）

  手动触发对账
  POST /payments/reconciliation/manual
  说明：手动触发每日对账任务，立即执行差异检测

  支付状态同步
  POST /payments/:id/sync-status
  说明：手动同步单笔支付状态（调用微信查询接口）
  （注：批量同步状态由前端批量调用此接口实现）

  ---
  支付状态对账流程（兜底机制）

  ┌─────────────┐    创建订单     ┌─────────────┐
  │  客户下单   │ ─────────────→ │ 待支付订单  │
  └─────────────┘               └──────┬──────┘
                                       │
                          ┌────────────┴────────────┐
                          │                         │
                          ▼                        ▼
                  ┌──────────────┐       ┌──────────────────┐
                  │ 客户正常支付  │       │ 每5分钟对账任务   │
                  └──────┬───────┘       └────────┬─────────┘
                         │                        │
                         ▼                        ▼
                  ┌──────────────┐       ┌──────────────────┐
                  │ 微信回调通知  │       │ 查询微信侧状态   │
                  └──────┬───────┘       └────────┬─────────┘
                         │                        │
                         └──────────┬─────────────┘
                                    ▼
                          ┌──────────────────┐
                          │ SUCCESS?         │
                          └──────┬───────────┘
                                 │ 是
                                 ▼
                          ┌──────────────────┐
                          │ 同步更新订单状态  │
                          └──────────────────┘

  说明：
  - 正常路径：微信回调通知 → 更新状态（快速）
  - 兜底路径：5分钟对账任务 → 主动查询 → 同步更新（慢但可靠）
  - 双轨并行确保支付状态最终一致

  不完整性说明

  1. 对账数据源
     - 前端 Reconciliation.tsx 调用 /api/v1/reconciliation/simple 接口
     - 该接口在后端尚未实现（后端仅有 reconciliation.task.ts 的任务逻辑）
     - 前端当前使用请求失败时的降级逻辑（显示空数据）

  2. 差异记录持久化
     - 每日对账任务发现的差异记录仅写入日志
     - 未建立专门的对账差异表来持久化存储
     - 后端 markDiscrepancies() 方法中预留了建表代码但被注释

  3. 对账历史
     - ReconciliationManagement.tsx 中的对账历史为模拟数据
     - 后端未提供对账执行历史查询接口

  4. 凭证统计
     - 对账中心的"凭证统计"标签页为占位状态

  ---
  关键设计特点

  1. 双轨对账 — 高频增量对账（5 分钟级支付状态同步）+ 每日全量差异报告
  2. 三级风险 — 差异金额自动分级：高(>100元)、中(10~100元)、低(0.01~10元)
  3. 自动通知 — 发现差异自动发送管理员通知
  4. 手动兜底 — 支持手动触发对账和单笔支付状态同步
  5. 可疑检测联动 — 对账发现的异常可与可疑支付检测模块联动处理
