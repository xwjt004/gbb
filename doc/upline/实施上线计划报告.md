## 1. 项目当前状态概述

### 1.1 项目定位

GBB儿童摄影管理系统 — 面向儿童摄影门店的全栈管理平台，涵盖订单管理、套系管理、支付财务、库存供应链、CRM会员、营销自动化等核心业务。

- **架构**：前后端分离（NestJS 11 + React 18），PostgreSQL数据库，Redis缓存
- **前端**：Ant Design 5 + TypeScript + Vite 5，管理后台 SPA
- **后端**：NestJS 11 + Prisma 6 ORM，RESTful API，全局前缀 `/api/v1`
- **部署**：Docker Compose 多容器编排，Nginx 反向代理

## 2. 上线阻塞项（P0 — 必须完成）

以下项目为上线前必须解决的阻塞项，任一缺失均不建议部署到生产环境。

### 2.1 安全加固

| # | 项目 | 说明 | 预估工时 |
|---|------|------|----------|
| S1 | JWT Secret 生产环境强制配置 | 启动时校验 JWT_SECRET 是否为默认值 `change-me-in-production`，如是则抛出异常阻止启动 | 0.5h |
| S2 | CSP 安全头配置 | 启用 nginx.conf 中注释掉的 CSP 头，限制 script-src / style-src / img-src | 1h |
| S3 | HTTPS 证书与强制跳转 | 申请 SSL 证书（Let's Encrypt），配置 nginx HTTPS 443 监听 + HTTP→HTTPS 301 跳转 | 2h |
| S4 | WeChat 支付回调签名验证 | `payment.controller.ts` 中微信回调处理逻辑目前 `always true`，需实现 HMAC-SHA256 验签 | 4h |
| S5 | 敏感操作二次确认 | 退款、删除订单/用户、取消订单等操作增加管理员密码或验证码确认 | 8h |
| S6 | 文件上传安全加固 | 限制上传类型白名单（仅图片/PDF）、文件大小上限硬校验、文件名消毒 | 2h |
| S7 | 环境变量校验 | NestJS 启动时使用 Joi 或 class-validator 校验所有必需环境变量 | 2h |

### 2.2 测试验收

| T2 | 核心业务 E2E 测试增强 | 覆盖完整业务流程：下单→支付→排片→完成 | 16h |

## 5. 上线实施计划

### 5.1 建议上线策略：分阶段灰度上线

```
第一阶段：最小可行产品上线（MVP Launch）
  ├── 核心功能：订单、套系、支付、用户、商品/库存
  ├── 仅开放管理员后台，不开放客户端
  └── 目标：验证系统稳定性与数据准确性

第二阶段：全功能上线（Full Launch）
  ├── 开放全部管理后台功能
  ├── 接入微信客户端（小程序/H5）
  └── 目标：替代现有管理流程

第三阶段：持续迭代（Iteration）
  ├── Phase 4 剩余功能
  ├── Phase 5 规划启动
  └── 目标：全面数字化
```

## 7. 风险与应急预案

### 7.1 已知风险

| # | 风险 | 影响 | 概率 | 应对方案 |
|---|------|------|------|----------|
| R1 | WeChat 支付配置复杂度高 | 支付功能不可用 | 高 | 第一阶段使用线下支付 + 手动确认替代，微信支付后期接入 |
| R2 | 备案等待时间长 | 上线延期 | 中 | 提前启动备案流程（建议立即提交），临时使用 IP 地址或海外服务器进行技术验证 |
| R3 | PostgreSQL 表数据量增长后性能下降 | 查询变慢 | 中 | 提前识别慢查询，建立索引策略；订单/日志等大表按月分区 |
| R4 | 首次部署环境差异导致容器启动失败 | 上线计划受阻 | 中 | 提前在预发布环境做完整部署演练 |
| R5 | 测试覆盖不足导致生产 Bug | 影响用户体验 | 高 | 灰度发布 + 逐步放量 + 实时监控 + 快速回滚能力 |
| R6 | 微信小程序管理端未开发，客户体验断层 | 客户无法线上自主操作 | 中 | 第一阶段定位为内部管理工具，客户交互沿用现有方式 |

### 7.2 应急预案

| 场景 | 响应措施 |
|------|----------|
| 应用服务宕机 | Docker restart policy（always）+ 健康检查自动恢复；5分钟内无法自愈则手动回滚至上一版本 |
| 数据库宕机/损坏 | 从每日备份恢复，目标 RTO < 4h，RPO < 24h |
| 支付异常（重复扣款/掉单） | 对账脚本（每日凌晨自动运行）标记差异项，人工介入处理 |
| 安全事件（数据泄露） | 立即下线服务 → 通知管理员 → 审计日志溯源 → 漏洞修复 → 恢复上线 |
| 版本回滚 | 保留上一版本的 Docker image tag，docker-compose 中切换 tag 即可回滚 |

### 7.3 回滚策略

```
# 后端回滚
docker pull your-registry/gbb-backend:previous-tag
docker-compose up -d backend

# 前端回滚（Nginx + 静态文件）
docker pull your-registry/gbb-frontend:previous-tag
docker-compose up -d frontend

# 数据库回滚（慎用！）
prisma migrate resolve --rolled-back <migration-name>
# 注：数据库回滚需人工介入数据修复，建议优先应用修复补丁而非回滚
```
