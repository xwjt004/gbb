# 乖宝宝儿童影楼团购设计方案
 数据模型设计、API 设计、核心业务流程、阶梯价格机制、优惠券策略、架构设计 
## 一、概述

团购（拼团）功能允许用户发起拼团活动，邀请好友共同购买套餐或商品，达到指定人数后可享受阶梯优惠价格。系统支持套餐团购和商品团购两种类型，以及阶梯价格体系。

### 1.1 核心目标

- 通过社交裂变（分享、拼团）提高订单转化率
- 以阶梯价格激励用户邀请更多人参与
- 成团后自动发放优惠券，促进复购

### 1.2 适用范围

- 微信小程序端：开团、参团、分享、支付
- 管理后台端：活动管理、阶梯价格配置、数据统计

## 二、数据模型设计

### 2.1 GroupBuyActivity（团购活动）

表名：`group_buy_activities`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String (UUID) | 主键 |
| packageId | Int? | 关联套餐（可选） |
| productId | Int? | 关联商品（可选） |
| creatorUserId | String | 团长（开团人） |
| minCount | Int (default 3) | 成团人数 |
| maxCount | Int? | 参团人数上限（null 为不限制） |
| status | VarChar(20) | 状态：ACTIVE / SUCCESS / FAILED |
| createdAt | DateTime | 创建时间 |
| expiredAt | DateTime | 过期时间（创建后 48 小时） |

关联关系：`Package?`, `Product?`, `WxUser`（团长）, `GroupBuyParticipant[]`, `Order[]`

### 2.2 GroupBuyParticipant（参团记录）

表名：`group_buy_participants`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | String (UUID) | 主键 |
| activityId | String | 关联活动 |
| userId | String | 参团用户 |
| status | VarChar(20) | JOINED / SUCCESS |
| joinedAt | DateTime | 加入时间 |

### 2.3 GroupBuyTier（阶梯价格）

表名：`group_buy_tiers`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Int (自增) | 主键 |
| packageId | Int? | 关联套餐 |
| productId | Int? | 关联商品 |
| minCount | Int | 达到该人数时适用此价格 |
| price | Decimal(10,2) | 阶梯价格 |
| createdAt | DateTime | 创建时间 |

索引：`[packageId, minCount]`, `[productId, minCount]`（联合唯一约束）

### 2.4 Package（套餐扩展字段）

| 字段 | 类型 | 说明 |
|------|------|------|
| groupMinCount | Int? | 成团所需最少人数 |
| groupPrice | Decimal? | 团购价（前端展示用） |
| groupBuyDescription | String? | 团购说明文字 |
| posterTitle | String? | 海报标题 |
| posterContent | String? | 海报描述 |
| posterBackground | String? | 海报背景色/图片URL |
| posterImages | String[] | 海报宣传图片 |

## 四、核心业务流程

### 4.1 开团流程

```
用户选择套餐/商品
  → 开团页设置成团人数
  → POST /group-buy/start
    → 校验：套餐/商品存在、用户无进行中团购
    → 创建 GroupBuyActivity（status = ACTIVE, expiredAt = now+48h）
    → 开团人自动成为第一个参团者（GroupBuyParticipant, status = JOINED）
  → 跳转团购详情页
```

### 4.2 参团流程

```
用户扫码/分享链接进入团购详情
  → 点击"参团"
  → POST /group-buy/:id/join
    → 校验：活动存在、活动未结束、未重复参团、未达人数上限
    → 创建参团记录
    → 检查人数是否 ≥ minCount → 是则调用 completeGroupBuy()
    → 计算当前阶梯价格
  → 进入预约流程（日期 → 时段 → 信息 → 确认）
  → 创建订单（自动关联 groupBuyActivityId）
  → 支付
```

### 4.3 成团逻辑

```
completeGroupBuy():
  → 更新活动 status = SUCCESS
  → 更新所有参团者 status = SUCCESS
  → 如为套餐团购：
    → 查找/创建 9 折优惠券模板（GROUP_BUY 类型）
    → 为每位参团者发放 UserCoupon（30天有效）
  → 发送开团成功通知（WECHAT 订阅消息）
```

### 4.4 过期处理

```
定时任务（每小时）：
  → 查询所有 status = ACTIVE 且 expiredAt < now 的活动
  → 标记为 FAILED
  → 对已支付的订单自动发起退款
```

### 4.5 退团流程

```
POST /group-buy/:id/leave
  → 校验：活动进行中、非团长、已参团
  → 删除参团记录
```

## 五、阶梯价格机制

### 5.1 匹配规则

- 查询与活动关联的套餐/商品的所有阶梯价格
- 按 minCount 降序排列
- 取第一个 participantCount ≥ tier.minCount 的阶梯价
- 如果没有匹配的阶梯价，返回 null（按标准价结算）

### 5.2 示例

| 参团人数 | 阶梯条件 | 适用价格 |
|---------|---------|---------|
| 2人 | minCount=2, ¥250 | ¥250 |
| 3人 | minCount=2, ¥250 / minCount=5, ¥200 | ¥250 |
| 5人 | minCount=5, ¥200 | ¥200 |

## 六、优惠券策略

成团后自动发放：

| 属性 | 值 |
|------|-----|
| 优惠券类型 | GROUP_BUY |
| 折扣类型 | PERCENT（百分比） |
| 折扣值 | 10（即 9 折） |
| 适用范围 | 限该套餐 |
| 有效期 | 30 天 |
| 防重复 | 检查 UserCoupon 是否已存在 |

## 七、订单关联

- 订单创建时自动查询用户参与的活动，将 `groupBuyActivityId` 关联到订单
- 后台订单列表和详情显示关联的团购活动（ID、状态、成团人数）
- 当前仅关联 ID，订单金额仍使用标准价（团购价尚未应用到订单金额）

## 九、技术实现要点

### 9.1 后端

- 框架：NestJS 11 + Prisma ORM 6 + PostgreSQL 17
- 模块位置：`src/modules/group-buy/`
- 定时任务：`@Cron(CronExpression.EVERY_HOUR)`
- 通知：NotificationsService（WECHAT 类型，失败不影响主流程）
- 支付退款：PaymentsService.processRefund()

### 9.2 前端管理后台

- 框架：React 18 + Ant Design 5 + Vite 5
- API 层：`src/services/groupBuy.ts`
- 数据导出：useExcelExport hook

### 9.3 小程序

- UI：Vant Weapp 组件库
- API：wx.request（JWT token 认证）
- 海报生成：Canvas 2D API（375×750px）
- 分享：onShareAppMessage / onShareTimeline

## 十、已知限制与待改进

1. **团购价未应用到订单金额**：参团用户支付标准价而非团购价，需在订单创建时替换价格
2. **退款未更新参团状态**：退款后 GroupBuyParticipant 仍为 SUCCESS，优惠券未回收
3. **后台创建订单未关联团购**：仅 wx-order 流程有关联逻辑
4. **阶梯价未在订单金额中体现**：前端获取阶梯价后未传递给订单创建接口
