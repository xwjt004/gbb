## 二、现有问题分析

### 2.1 功能重叠问题

| # | 问题 | 具体描述 |
|---|------|---------|
| P1 | **客户订单 vs 订单管理** | 「客户管理」分组下的子菜单"客户订单"直接跳转到 `/orders`，与独立的一级菜单「订单管理」指向同一页面，属于重复入口。 |
| P2 | **自动采购建议 vs 补货建议** | 采购管理下的"自动采购建议"(`/stock/auto-purchase`)在路由层被重定向到库存智能的"补货建议"(`/stock/restock-suggestions`)，说明两个入口指向同一功能，属于冗余。 |
| P3 | **顶部"功能"菜单完全重复** | 顶部"功能"下拉菜单中的 6 个入口（订单/套系/时间槽/支付/对账/员工）在侧边栏均有对应，纯属快捷冗余，与侧边栏无差异化定位。 |

### 2.2 分组不合理问题

| # | 问题 | 具体描述 |
|---|------|---------|
| G1 | **时间槽管理和拍摄日程割裂** | 两个页面强相关（一个设可用时段，一个排具体拍摄），却分成两个独立的一级菜单，不符合用户操作流程。 |
| G2 | **支付/对账/退款三权分立** | 支付管理、对账管理、退款管理三个一级菜单平级排列，实际属于同一业务链条（支付 → 对账 → 退款），分散后用户需跨菜单操作。 |
| G3 | **服务项目定位模糊** | 作为一级菜单只有单个页面，属于基础数据维护，与套系管理、商品管理关系紧密却独立在外。 |
| G4 | **DIY套系与套系管理分离** | DIY套系本质是套餐的一种灵活组合形式，逻辑上应隶属于套系管理，目前分作两个独立一级菜单。 |
| G5 | **采购管理与库存智能分离** | 采购(进)和库存(存)属于供应链上下游，目前分属两个一级菜单，且"自动采购建议"错放在采购侧实际功能在库存侧。 |

### 2.3 可发现性问题

| # | 问题 | 具体描述 |
|---|------|---------|
| D1 | **系统管理页面无侧边栏入口** | 系统设置、角色管理、操作日志、数据库备份/恢复等 11 个页面全部集中在顶部"系统"下拉菜单中，用户首次使用很难发现。 |
| D2 | **导出功能无侧边栏入口** | "导出"仅有顶部下拉菜单入口，侧边栏无对应。 |
| D3 | **通知管理无侧边栏入口** | 消息推送、通知模板等 5 个页面仅有顶部入口。 |
| D4 | **积分配置与积分明细归属不清** | 两个积分页面挂在"营销管理"下，但 PointsConfig 组件的源码文件位于 `System/` 目录，暗示历史归属存在分歧。 |

### 2.4 排序问题

| # | 问题 | 具体描述 |
|---|------|---------|
| S1 | **查找功能过早出现** | 「查找功能」(全局搜索)作为第 14 个一级菜单，位置靠前，但它属于工具类页面，使用频率低于业务页面，应置后。 |
| S2 | **菜单顺序与业务流程不符** | 当前顺序：客户→订单→套系→商品，实际业务流程是：先有套系/商品，再有订单，再到拍摄和支付。 |
| S3 | **数据分析位置靠后** | 作为管理决策入口，数据分析靠后于采购/库存等执行层功能，顺序可提前。 |

## 三、建议调整方案

### 3.1 核心原则

1. **按业务生命周期排序**：套系/商品（产品）→ 订单（销售）→ 拍摄/排片（交付）→ 支付/退款（财务）→ 营销（增长）
2. **同类合并、异类分离**：强相关功能合并为一个分组，弱相关功能拆开
3. **层级扁平化**：超过 6 个子页面时考虑拆分，单页面尽量归入相关分组
4. **可发现性优先**：所有页面必须在侧边栏有入口，顶部菜单仅作为快捷方式

### 3.2 推荐菜单结构

#### 第一组：经营概览

```
仪表盘
├── 概览
└── 数据看板
```

> 调整说明：保持不变，作为登录首页。

---

#### 第二组：产品中心

```
产品中心
├── 套系列表
├── 套系分类
├── DIY套系
│   ├── DIY套系列表
│   ├── DIY套系构建器
│   └── DIY折扣规则
├── 商品列表
├── 商品分类
└── 服务项目
```

> **调整说明**：将原来的「套系管理」「商品管理」「DIY套系」「服务项目」四个一级菜单合并为一个「产品中心」。为什么这样做——因为"服务项目"单页面不该独占一级菜单；DIY套系本质是套餐的子集，放在一起避免用户在"套系"和"DIY套系"之间来回切换。

---

#### 第三组：销售与订单

```
订单管理
├── 订单列表
└── 团购订单
```

> **调整说明**：移除"客户订单"冗余入口；团购订单从营销管理移至此处，因为它本质是订单的一种特殊形态，运营人员在处理订单时也需查看团购单，放在营销侧会增加跳转成本。

---

#### 第四组：客户管理

```
客户管理
├── 客户信息
├── 会员等级
└── 客诉管理
```

> **调整说明**：移除"客户订单"重复入口。CRM 的职责是管"人"而非管"单"，看订单应去订单管理。

---

#### 第五组：拍摄交付

```
拍摄管理
├── 时间槽管理
└── 拍摄日程
```

> **调整说明**：将原来的「时间槽管理」「拍摄日程」合并为一个分组。排片流程总是先设时间槽再排日程，放在一起符合操作习惯。

---

#### 第六组：财务管理

```
财务管理
├── 支付列表
├── 可疑支付检测
├── 对账管理
├── 退款审批
└── 退款记录
```

> **调整说明**：将「支付管理」「对账管理」「退款管理」三个一级菜单合并为「财务管理」。这三者处于同一资金链条：收款→对账→退款，任何环节出问题都要跨菜单查，合并后一站式处理。

---

#### 第七组：营销增长

```
营销管理
├── 优惠券管理
├── 积分配置
├── 积分明细
├── 客户分群
├── 营销活动
└── 数据分析
```

> **调整说明**：将「数据分析」从一级菜单移入营销管理。原因：数据分析当前的报表维度（转化率、客单价、复购率等）均为营销指标，放在此处上下文更匹配；同时腾出一级菜单位置。「团购管理」移至订单管理。

---

#### 第八组：供应链与库存

```
供应链管理
├── 供应商列表
├── 黑名单供应商
├── 采购订单
├── 采购审批
├── 在途商品
├── 入库管理
└── 库存智能
    ├── 销量预测
    ├── 安全库存
    ├── 补货建议
    ├── 呆滞预警
    └── 周转分析
```

> **调整说明**：将「采购管理」和「库存智能」合并为「供应链管理」。「库存智能」作为二级分组保留。移除"自动采购建议"冗余入口。这样供应链全链路（供应商→采购→在途→入库→库存→补货预测）在一个菜单下完成。

---

#### 第九组：系统管理

```
系统管理
├── 员工列表
├── 角色管理
├── 系统设置
├── 店铺信息
├── 打印设置
├── 自动化规则
├── 操作日志
├── 系统状态
├── 数据库备份
└── 数据库恢复
```

> **调整说明**：将原「员工管理」和顶部"系统"菜单中的页面全部移至侧边栏，新增「系统管理」分组。解决可发现性问题——所有系统级功能统一入口。

---

#### 第十组：工具

```
工具
├── 全局搜索
├── 导出中心
│   ├── 订单导出
│   ├── 员工导出
│   ├── 财务导出
│   └── 全部导出
└── 通知管理
    ├── 消息推送
    ├── 邮件发送
    ├── 系统通知
    ├── 库存报警
    └── 通知模板
```

> **调整说明**：将「查找功能」改为"全局搜索"，归入工具分组。将顶部菜单中的"导出"和"通知"下拉菜单改为侧边栏可访问。工具类功能使用频率不固定，统一放在最后不干扰主业务流程。

---

### 3.3 调整前后对比

| 调整前 | 调整后 |
|--------|--------|
| 18 个一级菜单 | 10 个一级菜单（分组更合理） |
| 4 个顶部菜单含系统级页面 | 顶部保留"工具"快捷入口 |
| 侧边栏 28 个可达页面 | 侧边栏 35+ 页面全覆盖 |
| 2 处重复入口（客户订单、自动采购建议） | 0 处重复入口 |
| 11 个系统页面仅顶部可访问 | 所有页面侧边栏可见 |

### 3.4 新菜单快捷入口建议

顶部菜单可简化为两个下拉：

| 快捷菜单 | 包含内容 |
|---------|---------|
| **快捷** | 高频操作：订单列表、客户查询、今日日程、新建团购 |
| **工具** | 导出中心、通知管理、全局搜索、计算器 |

移除原"系统"菜单（已移至侧边栏），移除原"功能"菜单（与侧边栏完全重复）。

## 四、实施建议

### 4.1 实施步骤

1. **调整菜单配置**：修改 `Layout/index.tsx` 中的 `menuItems` 数组，按新结构重组
2. **更新路由映射**：修改顶部快捷菜单和 `handleTopMenuClick` 逻辑
3. **移动组件文件**（可选）：将 PointsConfig 从 `System/` 移至 `Marketing/`，保持源码目录与菜单一致
4. **验证权限**：确认角色权限与新菜单结构的兼容性
5. **灰度发布**：先对测试环境部署，收集反馈后上线

### 4.2 代码变更范围

| 文件 | 变更内容 |
|------|---------|
| `Layout/index.tsx` | 重写 `menuItems` 数组、更新 `getTitleFromPath`、调整顶部菜单 |
| `App.tsx` | 无需变更（路由路径不变，仅菜单重组） |
| `pages/System/PointsConfig.tsx` | 可选：移至 `pages/Marketing/` |

### 4.3 风险与注意事项

- **用户习惯**：老用户已习惯现有菜单，建议在系统设置中保留"经典菜单"切换选项（为期 1-2 个月）
- **权限系统**：菜单结构变化不影响后端权限校验，前端仅做展示层重组
- **书签/链接**：所有页面路径不变，用户收藏的链接不会失效
- **顶部菜单过渡**：顶部"系统"菜单移除前，需确保所有管理员知晓侧边栏入口位置
