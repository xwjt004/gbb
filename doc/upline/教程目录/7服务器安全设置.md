如何让服务器更安全，不被侵入                                                                                                                 

● 基于之前的攻击教训（PostgreSQL 端口暴露导致勒索病毒），以下是针对您当前部署最有效的加固措施：
                         
  紧急/高优先级                     
                       
  1. SSH 仅密钥登录（当前可能已是，但需确认）
                            
  sudo sed -i 's/^#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
  sudo sed -i 's/^PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
  sudo systemctl restart sshd                                                                                                                    这杜绝了密码暴力破解。之后只能通过 SSH 密钥登录。
                                                                                                                                                 2. 安装 Fail2Ban 防暴力破解                                                         

  sudo apt install -y fail2ban
  sudo systemctl enable --now fail2ban
  自动封禁多次登录失败的 IP。

  3. Docker Compose 中的数据库密码

  当前 docker-compose.yml 里 PostgreSQL 密码是明文的。虽然端口已不对外暴露，但仍建议用 .env 管理所有敏感凭据：

  创建 /opt/gbb/.env 包含：
  POSTGRES_PASSWORD=baby_photo_2024
  REDIS_PASSWORD=gbb_redis_2024
  DATABASE_URL=postgresql://baby_photo:baby_photo_2024@postgres:5432/baby_photo_db
  JWT_SECRET=...

  然后在 compose 中引用 ${POSTGRES_PASSWORD} 变量。这样密码不会在 YAML 里明文存储（如果提交 Git）。

  4. 自动安全更新

  sudo apt install -y unattended-upgrades
  sudo dpkg-reconfigure --priority=low unattended-upgrades
  确保系统安全补丁自动安装。

  中优先级

  5. Docker 容器只读文件系统

  # 在 backend 服务中添加
  security_opt:
    - no-new-privileges:true
  read_only: true
  tmpfs:
    - /tmp

  6. 审计监控（检测异常登录）

  sudo apt install -y auditd
  sudo auditctl -w /etc/passwd -p wa -k passwd_changes
  sudo auditctl -w /var/log/auth.log -p wa -k auth_log

  7. 应用层 JWT 令牌

  当前 admin token 是 admin-token-${Date.now()} 简单格式。建议改用真正的 JWT（代码中已有 JWT_SECRET 环境变量）。

  长期/低优先级

  8. 腾讯云安全组（额外防护层）

  在腾讯云控制台设置安全组规则，与服务器 UFW 双重保障：仅允许 22/80/443 入站。

  9. Docker 镜像漏洞扫描

  定期用 docker scout 或 Trivy 扫描镜像中的已知漏洞。

  10. 数据库加密连接

  PostgreSQL 配置 SSL/TLS 连接，防止传输层嗅探。

  ---
  最关键的三件事（防御之前同类攻击）：防火墙已开 ✅ → SSH 仅密钥 + Fail2Ban → 敏感凭据移入 .env 文件。需要我帮您逐一执行吗？

  ### Steps 8-10 需要手动操作：

  Step 8 — 腾讯云安全组： 登录腾讯云控制台 → 云服务器 → 安全组，添加入站规则仅允许 22/80/443。这与服务器 UFW 形成双重防护，即使 UFW
  被意外关闭，安全组仍会拦截。

       需要添加 3 条入站规则：                                
                         
  规则 1 — SSH                      
  - 类型: 自定义                    
  - 来源: 0.0.0.0/0（或改为你本地的公网 IP，这样只有你能 SSH）
  - 协议端口: TCP:22           
  - 策略: 允许                  
  - 备注: SSH
                                                                                                                                                 规则 2 — HTTP
  - 类型: 自定义                                                                                                                                 - 来源: 0.0.0.0/0                                                                   
  - 协议端口: TCP:80
  - 策略: 允许
  - 备注: HTTP

  规则 3 — HTTPS
  - 类型: 自定义
  - 来源: 0.0.0.0/0
  - 协议端口: TCP:443
  - 策略: 允许
  - 备注: HTTPS

  添加后还需要删除安全组中已有的默认规则（通常是 ALLOW 0.0.0.0/0 ALL PORTS
  这种放行所有端口的规则），否则等于没防护。完成后服务器的入站流量就只有 22/80/443 三个口子，与服务器内的 UFW 规则一致，双重保障。

已经添加的 3 条规则（22/80/443）正确。现在需要处理那条危险规则：
                         
  0.0.0.0/0 ALL 允许 — 这条放行了所有端口（1-65535），你加的前面 3 条规则等于白费。必须把它删掉或改成拒绝。
                                    
  操作：点击该行右侧的 "删除" 或 "移除" 按钮（这不会影响服务器连接，因为 SSH 已经单独放行了）。
                               
  删除后只剩 4 条规则：         
  - TCP:22 允许 — SSH
  - TCP:80 允许 — HTTP                                                                                                                           - TCP:443 允许 — HTTPS
  - ALL 拒绝 (IPv6) — 保持不动                                                                                                                                                                                                       
  这样就完成了。安全组和服务器 UFW 双重防护，只开 3 个端口。
