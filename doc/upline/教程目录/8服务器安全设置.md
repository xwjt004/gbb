## 安全威胁回顾

此前因 PostgreSQL 端口（5432）暴露到公网，导致数据库被勒索病毒攻击。以下是针对该教训所做的全面安全加固措施。

---

## 已完成的安全加固（Steps 1-7）

| 步骤 | 状态 | 说明 |
|------|------|------|
| 1. SSH 仅密钥登录 | ✅ | `PasswordAuthentication` 已关闭，仅密钥登录 |
| 2. Fail2Ban 防暴力破解 | ✅ | SSH 5 次失败 → 封禁 1 小时 |
| 3. Docker .env 管理凭据 | ✅ | 所有敏感凭据从 YAML 移入 .env（chmod 600） |
| 4. 自动安全更新 | ✅ | `unattended-upgrades` 已启用 |
| 5. 容器只读文件系统 | ✅ | `read_only` + `tmpfs` + `no-new-privileges` |
| 6. 审计监控 | ✅ | `auditd` 监控 passwd/auth.log |
| 7. JWT 令牌 | ✅ | admin 登录生成真实 JWT |

### Step 8 — 腾讯云安全组（已完成）

安全组规则仅放行 22/80/443 入站端口：

| 规则 | 协议端口 | 来源 | 策略 | 说明 |
|------|---------|------|------|------|
| SSH | TCP:22 | 0.0.0.0/0 | 允许 | SSH 远程管理 |
| HTTP | TCP:80 | 0.0.0.0/0 | 允许 | HTTP 重定向到 HTTPS |
| HTTPS | TCP:443 | 0.0.0.0/0 | 允许 | HTTPS 网站访问 |
| 全部端口 | ALL | 0.0.0.0/0 | **拒绝** | 拒绝其他所有入站流量 |

与服务器 UFW 双重防护，即使 UFW 被意外关闭，安全组仍会拦截。

### Step 10 — PostgreSQL SSL（可选）

如需配置数据库加密连接，需生成证书并挂载到 postgres 容器，然后将 `DATABASE_URL` 追加 `?sslmode=require`。

## 端口开放清单

当前服务器仅开放 3 个端口：

| 端口 | 协议 | 用途 | 备注 |
|------|------|------|------|
| 22 | TCP | SSH | 仅密钥登录 |
| 80 | TCP | HTTP | 301 重定向到 HTTPS |
| 443 | TCP | HTTPS | 网站访问 |

以下端口**不对外暴露**（仅 Docker 内网可访问）：
- 3000（后端 API，通过 Nginx 反向代理访问）
- 5432（PostgreSQL）
- 6379（Redis）

---

## 安全最佳实践总结

1. **SSH 密钥认证** — 已关闭密码登录，防止暴力破解
2. **Fail2Ban** — 自动封禁恶意 IP
3. **最小端口原则** — 仅开放 22/80/443
4. **容器安全** — 只读文件系统 + 无新权限
5. **凭据管理** — .env 文件 chmod 600，不提交 Git
6. **自动更新** — 安全补丁自动安装
7. **审计日志** — 记录敏感文件变更
