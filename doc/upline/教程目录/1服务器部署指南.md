## 1. 前置准备

### 1.1 所需材料

| 项目 | 说明 | 获取方式 |
|------|------|---------|
| 云服务器 | 1核2GB 以上配置 | RackNerd / 阿里云 / 腾讯云等 |
| 域名 | 已备案（国内）或无需备案（国外） | Cloudflare / 阿里云 / Namecheap |
| 微信小程序 AppID | 小程序唯一标识 | 微信公众平台 |
| 微信小程序 AppSecret | 小程序密钥 | 微信公众平台 |
| Git 仓库 | 项目代码 | GitHub / Gitee |

### 1.2 域名 DNS 配置

在域名管理后台（如 Cloudflare）添加两条 A 记录，将域名指向服务器 IP：

| 类型 | 名称 | 值 | 代理 |
|------|------|-----|------|
| A | `@` | `服务器IP` | 开启（橙色云） |
| A | `www` | `服务器IP` | 开启（橙色云） |

> **名词解释**  
> - **A 记录**：将域名指向一个 IPv4 地址，是最基础的 DNS 记录类型。  
> - **`@`**：代表根域名（如 `guaibaobao.net`）。  
> - **`www`**：代表 `www.guaibaobao.net` 子域名。  
> - **代理状态（橙色云）**：开启后流量经过 Cloudflare 的 CDN 和防护网络，可以隐藏真实服务器 IP、加速访问、防御 DDoS 攻击。关闭（灰色云）则为纯 DNS 解析，流量直连服务器。

## 2. 服务器初始化

### 2.1 首次连接服务器

```bash
ssh root@服务器IP
```

> **命令说明**  
> `ssh` 是 Secure Shell 的缩写，用于安全地远程连接服务器。  
> `root` 是服务器的超级管理员账户，拥有最高权限。  
> 首次连接会提示确认主机指纹，输入 `yes` 即可。

### 2.2 配置 SSH 密钥登录（推荐）

在本地机器（你的电脑）上执行：

```bash
# 生成 SSH 密钥对（如果还没有）
ssh-keygen -t ed25519 -C "your-email@example.com"

# 将公钥复制到服务器
ssh-copy-id root@服务器IP
# 或者手动上传：cat ~/.ssh/id_ed25519.pub 然后粘贴到服务器的 ~/.ssh/authorized_keys
```

> **为什么这么做？**  
> SSH 密钥登录比密码登录更安全且更方便。密钥对由两部分组成：  
> - **私钥**（`id_ed25519`）：保存在本地，相当于你的"身份证"，绝不能泄露。  
> - **公钥**（`id_ed25519.pub`）：放在服务器上，用来验证你的身份。  
>   
> 配置好后登录不再需要输入密码，且黑客无法通过暴力破解密码入侵。

为了方便后续操作，可以在本地 `~/.ssh/config` 中添加服务器别名：

```bash
Host gbb-server
  HostName 服务器IP
  User root
  IdentityFile ~/.ssh/id_ed25519
  StrictHostKeyChecking no
```

之后就可以用 `ssh gbb-server` 快速连接了。

### 2.3 安装 Docker

```bash
# 官方一键安装脚本（推荐）
curl -fsSL https://get.docker.com | sh

# 验证安装是否成功
docker --version          # 查看 Docker 版本号
docker compose version    # 查看 Docker Compose 版本号
```

> **Docker 是什么？**  
> Docker 是一种容器技术，可以将应用及其依赖打包在"集装箱"（容器）中运行。  
> 好处是：无论在哪台服务器上运行，环境都是一致的，不会出现"我本地能跑，服务器跑不了"的问题。  
>   
> **Docker Compose 是什么？**  
> 用于定义和运行多个 Docker 容器的工具。本项目需要运行 5 个服务（后端、前端、数据库、缓存、反向代理），用 Compose 一条命令就能全部启动。

### 2.4 验证 Docker 运行

```bash
docker info
```

如果看到 `Server Version:` 等输出，说明 Docker 已正常运行。

## 3. 部署项目代码

### 3.1 创建项目目录

```bash
mkdir -p /opt/baby-photo
cd /opt/baby-photo
```

> **`/opt/baby-photo`**  
> `/opt` 是 Linux 中存放第三方软件的约定目录（optional 的缩写）。  
> 你也可以选择其他位置，但建议放在 `/opt` 下，符合 Linux 目录规范。

### 3.2 获取代码

方式一：从 Git 仓库克隆（推荐）

```bash
git clone <你的仓库地址> .
```

> **命令说明**  
> `git clone` 会下载整个仓库到当前目录。最后的 `.` 表示下载到当前目录而不是新建子目录。

方式二：从本地打包上传

```bash
# 在本地打包项目（排除不需要的文件）
tar czf baby-photo.tar.gz \
  --exclude='node_modules' \
  --exclude='dist' \
  --exclude='.git' \
  --exclude='.env' \
  -C /path/to/project .

# 上传到服务器
scp baby-photo.tar.gz root@服务器IP:/opt/baby-photo/

# 在服务器解压
cd /opt/baby-photo && tar xzf baby-photo.tar.gz
```

> **`tar` 命令参数说明**  
> - `c`：创建归档文件（create）  
> - `z`：通过 gzip 压缩（压缩比高）  
> - `f`：指定归档文件名  
> - `x`：解压（extract）  
>   
> **`--exclude`**：排除不需要上传的目录，减少传输时间和大小。  
> `node_modules` 和 `dist` 会在 Docker 构建时重新生成，`.git` 是版本控制文件，生产环境不需要。

## 4. 环境变量配置

### 4.1 创建 .env 文件

```bash
cd /opt/baby-photo
cp .env.production.example .env
vim .env     # 或使用 nano .env
```

> **什么是 .env 文件？**  
> `.env` 是环境变量配置文件，存放数据库密码、API 密钥等敏感信息。  
> 它不会被提交到 Git 仓库，需要手动创建。项目提供了 `.env.production.example` 作为模板。

### 4.2 必填配置项

以下变量 **必须设置**，否则应用无法启动：

```bash
# === 数据库 ===
DATABASE_URL="postgresql://postgres:postgres@postgres:5432/baby_photo_db"
# 数据库连接地址。格式：数据库类型://用户名:密码@主机:端口/数据库名
# 在 Docker 环境中，主机名填 Docker 服务名（如 postgres）
# 如果是远程数据库，填 IP 地址

# === JWT 密钥（务必修改为随机字符串） ===
JWT_SECRET="请修改为一个随机的长字符串，至少32位"
# JWT（JSON Web Token）用于用户登录后的身份验证。
# 此密钥用于签发和验证登录令牌，如果泄露，攻击者可以伪造任意用户登录。
# 生成方法：openssl rand -base64 32

# === 前端域名（CORS） ===
CORS_ORIGIN="https://guaibaobao.net"
# CORS（跨域资源共享）控制哪些域名可以访问后端 API。
# 多个域名用逗号分隔，如："https://a.com,https://b.com"

# === 微信小程序 ===
WX_APP_ID="你的小程序AppID"       # 微信公众平台 → 开发 → 基本配置
WX_APP_SECRET="你的小程序AppSecret" # 微信公众平台 → 开发 → 基本配置（需重置后复制）
# 这两个是微信小程序的身份凭证，AppID 是公开的，AppSecret 必须保密。

# === 微信支付 ===
WECHAT_MCH_ID="你的商户号"        # 微信支付商户平台的商户号
WECHAT_API_KEY="你的API密钥"      # 微信支付商户平台 → API安全 → 设置密钥
# 如果暂时没有微信支付，可以留空或填占位符，但支付功能不可用。

# === 加密密钥 ===
ENCRYPT_KEY="请修改为一个随机密钥"
# 用于加密敏感数据（如手机号等个人隐私信息）。
```

## 5. SSL/HTTPS 证书

### 5.1 方案一：Cloudflare SSL（推荐，最简单）

如果域名使用了 Cloudflare 代理（DNS 橙色云），可以：

1. 登录 Cloudflare 控制面板 → SSL/TLS
2. 将 SSL 加密模式改为 **Full**（全程加密）或 **Full (Strict)**（需源站有效证书）
3. 如果选择 **Full**，服务器可以使用自签名证书。Cloudflare 到用户浏览器使用标准 HTTPS 证书（自动管理），Cloudflare 到服务器使用你的自签名证书。

在服务器上生成自签名证书：

```bash
mkdir -p /opt/baby-photo/ssl

# 生成有效期 365 天的自签名证书
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /opt/baby-photo/ssl/key.pem \
  -out /opt/baby-photo/ssl/cert.pem \
  -subj "/CN=guaibaobao.net" \
  -addext "subjectAltName=DNS:guaibaobao.net,DNS:www.guaibaobao.net"
```

> **参数说明**  
> - `req -x509`：创建自签名证书（不是由 CA 签发）  
> - `-nodes`：不加密私钥（no DES），这样 nginx 启动时不需要输入密码  
> - `-days 365`：有效期 365 天  
> - `-newkey rsa:2048`：生成 2048 位的 RSA 密钥（安全性足够且性能好）  
> - `-subj`：证书所属信息，`/CN=` 后填域名  
> - `-addext`：添加 Subject Alternative Name（SAN），现代浏览器要求此字段包含实际域名

### 5.2 方案二：Let's Encrypt 免费证书（需公网 IP 且 80 端口可达）

```bash
# 安装 Certbot
apt install -y certbot    # Ubuntu/Debian
yum install -y certbot    # CentOS

# 申请证书（需 80 端口可访问）
certbot certonly --standalone -d guaibaobao.net -d www.guaibaobao.net

# 证书文件位置
ls /etc/letsencrypt/live/guaibaobao.net/
# 会生成：cert.pem（域名证书）、chain.pem（中间证书）、fullchain.pem（完整链）、privkey.pem（私钥）

# 复制到项目目录
cp /etc/letsencrypt/live/guaibaobao.net/fullchain.pem /opt/baby-photo/ssl/cert.pem
cp /etc/letsencrypt/live/guaibaobao.net/privkey.pem /opt/baby-photo/ssl/key.pem

# 设置自动续期（Let's Encrypt 证书有效期 90 天）
echo "0 3 * * * certbot renew --quiet && docker exec gbb-nginx nginx -s reload" | crontab -
```

> **什么是 Let's Encrypt？**  
> Let's Encrypt 是一个免费的证书颁发机构（CA），提供自动化的 SSL 证书申请和续期。  
> 它的证书有效期为 90 天，需要通过 crontab 定期续期。  
>   
> **`crontab` 语法**：`分 时 日 月 周 命令`  
> `0 3 * * *` 表示每天凌晨 3 点执行。`--quiet` 表示没有成功消息时不输出日志。

## 6. 构建与启动

### 6.1 项目结构说明

本项目的 Docker 部署包含以下服务（定义在 `docker-compose.yml` 中）：

| 服务名 | 容器名 | 功能 | Dockerfile 位置 |
|--------|--------|------|-----------------|
| `postgres` | gbb-postgres | PostgreSQL 数据库 | 官方镜像 |
| `redis` | gbb-redis | Redis 缓存（可选） | 官方镜像 |
| `backend` | gbb-backend | NestJS 后端 API | `server/baby-photo-backend/Dockerfile` |
| `admin-frontend` | gbb-admin-frontend | 管理后台前端 | `client/admin-frontend/Dockerfile` |
| `nginx` | gbb-nginx | 反向代理 | 官方镜像 + 自定义配置 |
| `prisma-studio` | gbb-prisma-studio | 数据库管理界面（仅开发） | 使用 `base` 阶段 |

### 6.2 构建镜像

```bash
cd /opt/baby-photo

# 构建所有服务的镜像
docker compose build

# 也可以单独构建某个服务（修改代码后只需要重建改动的服务）
docker compose build backend
```

> **`docker compose build` 在做什么？**  
> 它读取每个服务的 `build` 配置，按照 Dockerfile 的指令一步步执行：  
> 1. 拉取基础镜像（如 `node:20-alpine`）  
> 2. 安装系统依赖  
> 3. 复制代码到镜像  
> 4. 安装 npm 依赖（`npm ci`）  
> 5. 编译 TypeScript（`npm run build`）  
> 6. 生成 Prisma 客户端  
> 7. 创建运行用户、设置权限  
> 8. 将最终产物打包成镜像

#### Dockerfile 多阶段构建详解

以后端 Dockerfile 为例，它包含 3 个阶段：

```dockerfile
# Stage 1: base — 安装所有依赖
FROM node:20-alpine AS base
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci                    # npm ci 比 npm install 更快更安全，会锁定版本
COPY . .

# Stage 2: builder — 编译 TypeScript 为 JavaScript
FROM base AS builder
ENV NODE_OPTIONS="--max-old-space-size=2048"  # 防止 2GB 内存的 VPS 在构建时 OOM
RUN npx prisma generate && npm run build

# Stage 3: runner — 精简的生产运行镜像
FROM node:20-alpine AS runner
# ... 只复制 dist（编译产物）和 node_modules，不包含源码
```

> **为什么需要多阶段构建？**  
> 1. **减小镜像体积**：最终镜像只包含运行所需的文件（编译后的 JS），不包含 TypeScript 源码和构建工具。  
> 2. **提高安全性**：源码和构建工具不会暴露在生产镜像中。  
> 3. **加快部署**：小镜像传输更快。  
>   
> **`--max-old-space-size=2048` 的作用**：  
> Node.js 默认内存限制约 1.4GB（V8 引擎限制）。1-2GB 内存的 VPS 在编译大型 TypeScript 项目时容易内存溢出（OOM），此参数将 Node.js 堆内存上限提高到 2GB。

### 6.3 启动服务

首次启动会先构建镜像，然后创建并运行所有容器：

```bash
# 启动所有服务
docker compose up -d

# 只启动核心服务（不启动 nginx，等后端就绪后再启动）
docker compose up -d postgres redis backend

# 所有服务就绪后，再启动 nginx
docker compose up -d nginx
```

> **`-d` 参数**：daemon（守护进程）模式，容器在后台运行，不会占用终端。

### 6.4 服务的启动顺序依赖

```yaml
# docker-compose.yml 中定义的依赖关系
services:
  backend:
    depends_on:
      postgres:
        condition: service_healthy  # 等待数据库健康后再启动后端
      redis:
        condition: service_healthy

  nginx:
    depends_on:
      - backend                     # nginx 依赖后端（但不会阻塞等待）
```

> **`depends_on` 与 `condition: service_healthy`**  
> - `depends_on` 告诉 Docker Compose 服务的启动顺序。  
> - `condition: service_healthy` 确保上一个服务的健康检查通过后，才启动当前服务。  
> - 健康检查（healthcheck）会定期执行指定命令（如 `pg_isready` 检查数据库是否可连接），只有返回 0（成功）才算健康。

### 6.5 验证服务状态

```bash
# 查看所有容器状态
docker ps

# 会显示类似以下信息：
# NAMES                STATUS
# gbb-backend          Up 5 minutes (healthy)
# gbb-postgres         Up 5 minutes (healthy)
# gbb-redis            Up 5 minutes (healthy)
# gbb-nginx            Up 5 minutes
# gbb-admin-frontend   Up 5 minutes
```

> **`docker ps` 输出的 STATUS 列含义**  
> - `Up 5 minutes`：已运行 5 分钟  
> - `healthy`：健康检查通过（仅定义了 healthcheck 的服务会显示）  
> - `unhealthy`：健康检查失败  
> - `Restarting (1)`：正在重启，括号内为重启次数  
> - `health: starting`：还在启动阶段，尚未执行首次健康检查

### 6.6 重启服务

修改代码或配置后，需要重建并重启：

```bash
# 完整流程：停止 → 删除旧容器 → 重建镜像 → 重新创建并启动
docker stop gbb-backend
docker rm gbb-backend
docker compose build backend
docker compose up -d backend

# 或者用更简洁的方式（Docker Compose 会自动处理）
docker compose up -d --build backend

# 重启 nginx 加载新配置（不需要重建，只是重启容器）
docker compose up -d nginx
```

> **为什么用 `stop + rm` 而不是直接 `up -d`？**  
> 当镜像有更新时，`docker compose up -d` 会自动重建容器。但如果容器名和网络配置有变动，或者你想确保使用全新的容器，先删除再创建更干净。

## 7. 数据库迁移

### 7.1 自动迁移

后端容器启动时，`docker-entrypoint.sh` 会自动执行数据库迁移：

```bash
#!/bin/sh
set -e  # 任何命令失败就退出脚本

echo "=== Running database migrations ==="
npx prisma migrate deploy 2>&1   # 将 Prisma 模型变更应用到数据库

echo "=== Starting application ==="
exec node dist/main.js           # 启动应用（exec 替换当前进程，确保信号正确处理）
```

> **`prisma migrate deploy` vs `prisma migrate dev`**  
> - `migrate dev`：**开发环境**使用，会自动创建新的迁移文件并应用。  
> - `migrate deploy`：**生产环境**使用，只执行尚未应用的迁移文件，不会创建新文件，更安全。  
> - 两者都在 `prisma/migrations/` 目录下查找迁移文件。

### 7.2 手动执行迁移

如果自动迁移失败，可以进入容器手动执行：

```bash
# 进入正在运行的后端容器
docker exec -it gbb-backend /bin/sh

# 在容器内执行迁移
cd /app
npx prisma migrate deploy

# 查看迁移状态
npx prisma migrate status

# 退出容器
exit
```

> **`docker exec -it` 参数**  
> - `exec`：在运行中的容器内执行命令  
> - `-i`：交互模式（interactive）  
> - `-t`：分配终端（tty）  
> 合起来就是"以交互终端方式进入容器"。

### 7.3 修复失败的迁移

如果迁移中途失败导致数据库状态不一致，可以手动修复：

```bash
# 进入 PostgreSQL 容器
docker exec -it gbb-postgres psql -U postgres -d baby_photo_db

# 查看迁移记录表
SELECT * FROM _prisma_migrations;

# 将失败的迁移标记为已回滚
UPDATE _prisma_migrations
SET rolled_back_at = NOW()
WHERE migration_name = '失败的迁移文件名';

# 手动创建缺失的表或枚举类型
CREATE TABLE IF NOT EXISTS ...;
CREATE TYPE "EnumName" AS ENUM ('VALUE1', 'VALUE2');
ALTER TABLE ... ALTER COLUMN ... TYPE "EnumName" USING (...::text::"EnumName");

# 将迁移标记为已完成
UPDATE _prisma_migrations
SET finished_at = NOW(), rolled_back_at = NULL
WHERE migration_name = '失败的迁移文件名';
```

> **`_prisma_migrations` 表**：Prisma 用这个表跟踪哪些迁移已经执行过。如果迁移脚本本身有问题（比如创建了数据库中已存在的枚举），可以手动修复后更新此表的状态。

### 7.4 手动创建缺失的表

如果某个表在迁移中被遗漏（例如后来新增的 model 但没有生成对应的迁移文件）：

```bash
# 根据 Prisma schema 手动创建表
docker exec gbb-postgres psql -U postgres -d baby_photo_db -c "
CREATE TABLE IF NOT EXISTS public.member_levels (
    id SERIAL PRIMARY KEY,
    level INTEGER NOT NULL UNIQUE,
    name VARCHAR(50) NOT NULL,
    min_growth INTEGER NOT NULL,
    max_growth INTEGER NOT NULL,
    benefits JSONB,
    icon VARCHAR(255),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
"
```

> **为什么会出现缺少表的情况？**  
> Prisma schema 中新增了 model（如 `MemberLevel`），但对应的迁移文件没有正确生成或执行。  
> 这可能是因为迁移文件被手动标记为已完成但表创建 SQL 未执行。添加 `CREATE TABLE IF NOT EXISTS` 可以安全地补建表。

## 8. 常见问题排查

### 8.1 后端 Crash Loop（反复重启）

**现象**：`docker ps` 显示后端不断在 `Restarting`

**排查步骤**：

```bash
# 1. 查看日志
docker logs gbb-backend --tail 50

# 2. 常见原因及解决方案
```

**原因①：数据库表不存在**

```
Error: Invalid `prisma.memberLevel.count()`:
The table `public.member_levels` does not exist in the current database.
```

**解决方案**：手动创建缺少的表（参考 7.4 节）。

---

**原因②：环境变量验证失败**

```
Error: JWT_SECRET 仍为默认值
```

**解决方案**：在 `.env` 中修改 `JWT_SECRET` 为随机字符串。

---

**原因③：Node.js 内存不足（OOM）**

构建时（`docker compose build`）出现：

```
FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed
```

**解决方案**：在 Dockerfile 构建阶段添加环境变量：

```dockerfile
ENV NODE_OPTIONS="--max-old-space-size=2048"
```

---

### 8.2 nginx 启动失败

**现象**：nginx 容器不断重启

**排查**：

```bash
docker logs gbb-nginx
```

**常见原因**：nginx 配置中引用了不存在的上游服务

```
[emerg] host not found in upstream "prisma-studio:5555"
```

**解决方案**：删除 nginx.conf 中未使用的 `upstream` 块，或确保引用的服务已启动。

---

### 8.3 健康检查失败

**现象**：后端状态显示 `(unhealthy)`，nginx 也反复重启

**排查**：

```bash
# 查看健康检查详情
docker inspect --format='{{json .State.Health}}' gbb-backend

# 检查健康检查命令是否正确
# 错误示例（CMD 格式导致 || 被当作参数传给 wget）：
test: ["CMD", "wget", "...", "||", "exit", "1"]
# 正确示例（CMD-SHELL 格式，整个命令作为 shell 执行）：
test: ["CMD-SHELL", "wget ... || exit 1"]
```

**常见错误**：`CMD` exec 格式会把 `||` `exit` `1` 作为 wget 的参数，wget 会尝试解析 `||` 为主机名导致超时。

**解决方案**：使用 `CMD-SHELL` 格式，让字符串通过 shell 解释执行。

---

### 8.4 后端监听地址错误

**现象**：后端日志显示"启动成功"，但 nginx 连接后端时返回 502

**排查**：

```bash
# 检查后端日志中的监听地址
docker logs gbb-backend | grep 监听
# 如果显示：监听地址: 127.0.0.1 (仅本地)
# 说明后端只监听本地回环地址，其他容器无法访问
```

**原因**：生产环境配置将 NestJS 绑定到 `127.0.0.1`，但在 Docker 环境中，nginx 运行在另一个容器中，需要通过 Docker 网络访问后端。`127.0.0.1` 在每个容器中都是自己的本地回环地址，其他容器访问不到。

**解决方案**：改为监听 `0.0.0.0`（所有网络接口）：

```typescript
// src/main.ts
const listenHost = '0.0.0.0';  // 不再区分开发/生产，Docker 下统一用 0.0.0.0
await app.listen(port, listenHost);
```

> **`0.0.0.0` 与 `127.0.0.1` 的区别**  
> - `127.0.0.1`：只监听本机回环接口，只有本容器内的进程能访问。  
> - `0.0.0.0`：监听所有网络接口（包括 Docker 网络），其他容器可以通过容器 IP 访问。  
> - 在 Docker 环境中，nginx 通过 Docker 内部网络连接后端，所以后端必须监听 `0.0.0.0`。

## 9. 运维命令速查

### 9.1 容器管理

```bash
# 查看所有运行中的容器
docker ps

# 查看所有容器（包括已停止的）
docker ps -a

# 查看指定容器的日志（最新 50 行）
docker logs gbb-backend --tail 50

# 实时跟踪日志输出（类似 tail -f）
docker logs -f gbb-backend

# 进入容器内部
docker exec -it gbb-backend /bin/sh

# 停止容器
docker stop gbb-backend

# 删除容器（会丢失未保存的数据）
docker rm gbb-backend

# 重启容器
docker restart gbb-backend
```

### 9.2 Docker Compose

```bash
# 构建所有镜像
docker compose build

# 构建并启动所有服务（后台运行）
docker compose up -d

# 启动指定服务
docker compose up -d backend nginx

# 停止所有服务
docker compose down

# 停止所有服务并删除数据卷（会丢失数据库数据！）
docker compose down -v

# 查看服务状态
docker compose ps

# 查看所有服务的日志
docker compose logs --tail 50
```

> **`docker compose down -v` 的严重后果**  
> `-v` 会删除所有 `volumes:` 中定义的数据卷，包括 PostgreSQL 的数据库文件。  
> 执行后所有数据将永久丢失！日常维护中**不要**使用 `-v` 参数。

### 9.3 数据库操作

```bash
# 进入 PostgreSQL 命令行
docker exec -it gbb-postgres psql -U postgres -d baby_photo_db

# 列出所有表
\dt

# 查看表结构
\d 表名

# 运行 SQL 文件
docker exec -i gbb-postgres psql -U postgres -d baby_photo_db < 文件.sql

# 备份数据库
docker exec gbb-postgres pg_dump -U postgres baby_photo_db > backup.sql

# 恢复数据库
cat backup.sql | docker exec -i gbb-postgres psql -U postgres -d baby_photo_db
```

> **`pg_dump`**：PostgreSQL 的备份工具，导出的 SQL 文件包含完整的建表和插数语句。备份是运维中最重要的事情之一！

### 9.4 查看资源使用

```bash
# 查看容器资源使用
docker stats

# 查看磁盘使用
df -h

# 查看内存使用
free -h

# 查看 Docker 磁盘占用
docker system df
```
