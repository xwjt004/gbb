## 2. 如何连接服务器

### 2.1 使用 SSH 密钥连接（推荐方式）

```bash
# 使用密钥登录
ssh ubuntu@81.68.236.12
```

> **注意**  
> 腾讯云服务器默认使用 `ubuntu` 用户（非 `root`）。如果需要执行特权命令，使用 `sudo`。

### 2.2 使用 SSH 别名连接（更快捷）

编辑本地 `~/.ssh/config` 文件，添加以下内容：

```ini
Host gbb-server
  HostName 81.68.236.12
  User ubuntu
  IdentityFile ~/.ssh/id_ed25519
```

添加后，只需要输入以下命令即可连接服务器：

```bash
ssh gbb-server
```

### 2.3 连接后的第一个检查

```bash
# 查看所有 Docker 容器状态
docker ps

# 正常应该看到类似这样（全部 Up 且 healthy）：
# NAMES                STATUS
# gbb-backend          Up 2 hours (healthy)
# gbb-postgres         Up 2 hours (healthy)
# gbb-redis            Up 2 hours (healthy)
# gbb-nginx            Up 2 hours
```

> **注意**：当前生产环境没有独立的 admin-frontend 容器，前端由 nginx 直接挂载静态文件提供。

```bash
# 查看所有容器（包括已停止的）

```

正常状态一览：

| 容器名 | 服务 | 镜像 | 必须运行？ | 正常状态 |
|--------|------|------|-----------|---------|
| `gbb-backend` | 后端 API | `baby-photo-backend:latest`（本地构建） | 是 | `Up (healthy)`（端口仅 Docker 内网，不对外暴露） |
| `gbb-postgres` | 数据库 | `postgres:17-alpine`（SSL 连接） | 是 | `Up (healthy)` |

### 后端容器安全配置

生产环境后端容器启用了以下安全措施：

| 配置 | 值 | 说明 |
|------|-----|------|
| `read_only` | `true` | 容器文件系统只读，无法写入任何文件 |
| `security_opt` | `no-new-privileges:true` | 禁止提权操作 |
| `tmpfs` | `/tmp`, `/app/temp`, `/app/backups` | 只读容器所需的临时可写目录 |
| Prisma 客户端 | 内置在 Docker 镜像中（非挂载） | 更新 Prisma schema 需要重建 Docker 镜像 |

```bash
# ===== 查看日志 =====
```

## 6. SSL/HTTPS 证书

### 6.1 当前使用的方案

我们使用 **Let's Encrypt** 免费证书，证书文件位于 `/opt/gbb/nginx/ssl/`：

```bash
ls -la /opt/gbb/nginx/ssl/
# fullchain.pem  — 完整证书链（公钥）
# privkey.pem    — 私钥文件（保密！不可泄露）
```

### 6.2 证书续期

Let's Encrypt 证书有效期 90 天，已配置 crontab 自动续期：

```bash
# 查看自动续期任务
sudo crontab -l

# 内容：
# 0 3 * * * certbot renew --quiet --post-hook "cp ... && docker exec gbb-nginx nginx -s reload"
```

如需手动续期：

```bash
# 手动续期
sudo certbot renew

# 复制新证书到 nginx 目录
sudo cp /etc/letsencrypt/live/guaibaobao.cn/fullchain.pem /opt/gbb/nginx/ssl/
sudo cp /etc/letsencrypt/live/guaibaobao.cn/privkey.pem /opt/gbb/nginx/ssl/

# 重启 nginx 加载新证书
docker exec gbb-nginx nginx -s reload
```

### 6.3 首次申请新域名的证书

```bash
# 安装 certbot
sudo apt install -y certbot

# 停止 nginx（防止 80 端口被占用）
docker stop gbb-nginx

# 申请证书
sudo certbot certonly --standalone -d guaibaobao.cn -d www.guaibaobao.cn

# 复制到 nginx 目录
sudo cp /etc/letsencrypt/live/guaibaobao.cn/fullchain.pem /opt/gbb/nginx/ssl/
sudo cp /etc/letsencrypt/live/guaibaobao.cn/privkey.pem /opt/gbb/nginx/ssl/

# 启动 nginx
docker compose up -d nginx

# 配置自动续期
sudo crontab -e
# 添加：0 3 * * * certbot renew --quiet --post-hook "cp /etc/letsencrypt/live/guaibaobao.cn/fullchain.pem /opt/gbb/nginx/ssl/ && cp /etc/letsencrypt/live/guaibaobao.cn/privkey.pem /opt/gbb/nginx/ssl/ && docker exec gbb-nginx nginx -s reload"
```

## 9. 构建与更新

### 9.1 更新后端（仅代码变更，无 Schema 变更）

后端采用 dist 挂载方式（`/opt/gbb/backend-dist:/app/dist`），TypeScript 编译后的 JS 文件直接从宿主机挂载到容器内，**无需重建 Docker 镜像**：

```bash
# 在本地开发机器执行
cd server/baby-photo-backend

# 第 1 步：编译 TypeScript
npx tsc

# 第 2 步：将 dist 上传到服务器
tar czf - -C dist . | ssh ubuntu@81.68.236.12 "sudo tar xzf - -C /opt/gbb/backend-dist && sudo chown -R ubuntu:ubuntu /opt/gbb/backend-dist"

# 第 3 步：重启容器加载新代码
ssh ubuntu@81.68.236.12 "docker restart gbb-backend"

# 第 4 步：确认服务正常
curl https://guaibaobao.cn/api/v1/health
```

### 9.2 更新后端（含 Prisma Schema 变更）

如果修改了 `prisma/schema.prisma`（新增/修改数据模型），需要**重建 Docker 镜像**，因为 Prisma Client 内置在镜像中，无法单独挂载。

**方式一：从本地构建并推送**

```bash
# 在本地执行（需要 Docker Desktop）
cd server/baby-photo-backend

# 第 1 步：构建 Docker 镜像
docker build -t baby-photo-backend:latest .

# 第 2 步：将镜像流式传输到服务器（无需中间文件）
docker save baby-photo-backend:latest | ssh ubuntu@81.68.236.12 "docker load"

# 第 3 步：将最新 dist 上传到服务器
npx tsc
tar czf - -C dist . | ssh ubuntu@81.68.236.12 "sudo tar xzf - -C /opt/gbb/backend-dist && sudo chown -R ubuntu:ubuntu /opt/gbb/backend-dist"

# 第 4 步：重启容器
ssh ubuntu@81.68.236.12 "docker stop gbb-backend && docker rm gbb-backend && cd /opt/gbb && docker compose up -d backend"

# 第 5 步：确认服务正常
curl https://guaibaobao.cn/api/v1/health
```

> **注意**：生产环境后端容器为 `read_only: true`，文件系统不可写。所有临时文件放在 `tmpfs` 挂载的 `/tmp`、`/app/temp` 目录中。

**方式二：在服务器上直接构建**

```bash
# 直接 SSH 到服务器执行
ssh ubuntu@81.68.236.12

# 项目代码需已在 /opt/gbb/ 目录下
cd /opt/gbb

# 构建镜像（使用 Dockerfile）
docker build -t baby-photo-backend:latest ./server/baby-photo-backend

# 重启服务
docker compose up -d backend
```

### 9.2 更新前端

前端不在服务器上构建，而是在本地构建后同步静态文件：

```bash
# 在本地开发机器执行
cd client/admin-frontend

# 构建（base 路径为 /admin/）
npm run build

# 同步到服务器（使用 tar + ssh 管道）
tar czf - dist/ | ssh ubuntu@81.68.236.12 "sudo tar xzf - -C /opt/gbb/www/admin && sudo chown -R ubuntu:ubuntu /opt/gbb/www/admin"

# 重启 nginx 使新文件生效（或直接刷新，nginx 会读取新文件）
ssh ubuntu@81.68.236.12 "docker restart gbb-nginx"
```

### 9.3 仅重启服务（不改代码）

```bash
# 重启所有服务
docker compose restart

# 重启单个服务
docker compose restart backend
```

### 9.4 完整重新部署

```bash
# 仅在极端情况下使用

# 1. 停止并删除所有容器
cd /opt/gbb
docker compose down

# 2. 清理旧镜像

docker compose build

# 4. 检查服务状态
```

## 10. 备份与恢复

### 10.1 手动备份

```bash
# 数据库备份（压缩）
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
docker exec gbb-postgres pg_dump -U baby_photo -d baby_photo \
  --create --clean --if-exists \
  --no-owner --no-privileges | gzip > \
  /opt/gbb/backups/db-backup-$TIMESTAMP.sql.gz

# 配置文件备份
tar czf /opt/gbb/backups/config-backup-$TIMESTAMP.tar.gz \
  /opt/gbb/docker-compose.yml \
  /opt/gbb/.env \
  /opt/gbb/nginx/

echo "备份完成: /opt/gbb/backups/db-backup-$TIMESTAMP.sql.gz"
ls -lh /opt/gbb/backups/
```

### 10.2 下载备份到本地

```bash
# 在本地机器执行
scp ubuntu@81.68.236.12:/opt/gbb/backups/db-backup-最新时间.sql.gz ./backups/
```

### 10.3 恢复数据库

```bash
# 解压备份文件
gunzip -c /opt/gbb/backups/db-backup-xxx.sql.gz > /tmp/restore.sql

# 复制到容器
docker cp /tmp/restore.sql gbb-postgres:/tmp/restore.sql

# 恢复（谨慎操作，会覆盖当前数据）
docker exec -i gbb-postgres psql -U baby_photo -d baby_photo < /tmp/restore.sql
```

### 10.4 备份文件清单

| 备份文件 | 内容 | 备份频率 |
|---------|------|---------|
| `db-backup-*.sql.gz` | PostgreSQL 数据库完整备份 | 建议每日 |
| `config-backup-*.tar.gz` | docker-compose.yml、.env、nginx 配置 | 配置变更时 |

## 11. 常见问题排查

### 11.1 网站打不开（502 Bad Gateway）

**检查步骤**：

```bash
# 第 1 步：检查所有容器是否正常运行
docker ps

# 第 2 步：检查后端日志
docker logs gbb-backend --tail 30

# 第 3 步：直接测试后端健康检查
curl https://guaibaobao.cn/api/v1/health

# 第 4 步：检查 nginx 日志
docker logs gbb-nginx --tail 20
```

### 11.2 后端不断重启（Crash Loop）

```bash
# 查看日志定位原因
docker logs gbb-backend --tail 50
```

**常见原因**：
- 数据库连接失败 → 检查 postgres 是否 healthy
- JWT_SECRET 未设置 → 检查 .env 文件
- TypeScript 编译问题 → 确认 Dockerfile 使用 `npx tsc` 而不是 `npm run build`（`module: "Node16"` 时 nest build 不生成 .js）

### 11.3 API 返回 400 Bad Request

```bash
# 后端日志会显示具体哪个接口报错
docker logs gbb-backend --tail 50 | grep 400
```

**常见原因**：后端 DTO 缺少 class-validator 装饰器。全局 `ValidationPipe` 配置了 `forbidNonWhitelisted: true`，没有 `@IsString()`、`@IsOptional()` 等装饰器的属性会被拒绝。

**解决方案**：给对应的 DTO 添加 class-validator 装饰器。

### 11.4 前端图片/资源加载失败

**现象**：页面能打开但图片不显示，控制台显示 `http://localhost:3000/...` 连接失败

**原因**：前端代码中硬编码了 `http://localhost:3000` 作为后端地址。

**解决方案**：已修复并部署。如果再次出现，检查 `src/utils/image.ts` 等文件中是否有 `localhost:3000` 硬编码。

### 11.5 SSL 证书到期

```bash
# 检查证书到期时间
sudo openssl x509 -in /opt/gbb/nginx/ssl/fullchain.pem -noout -dates

# 手动续期
sudo certbot renew
sudo cp /etc/letsencrypt/live/guaibaobao.cn/fullchain.pem /opt/gbb/nginx/ssl/
sudo cp /etc/letsencrypt/live/guaibaobao.cn/privkey.pem /opt/gbb/nginx/ssl/
docker exec gbb-nginx nginx -s reload
```

### 11.6 磁盘空间不足


## 12. 附录：Docker 入门

### 12.1 什么是 Docker？

Docker 是一种"容器"技术，可以理解为一个轻量级的虚拟机。

- **传统方式**：在服务器上直接安装 Node.js、PostgreSQL、Redis、nginx... 每个都要配置、版本冲突、卸载麻烦
- **Docker 方式**：每个应用打包在自己的"容器"里，互不干扰，一条命令启动所有服务

类比：
- **镜像（Image）** = 安装包
- **容器（Container）** = 安装好并正在运行的程序
- **Docker Compose** = 一键安装脚本（定义多个服务如何配合）

### 12.2 常用 Docker 命令速查

| 命令 | 作用 | 类比 |
|------|------|------|
| `docker ps` | 查看运行中的容器 | 查看正在运行的进程 |
| `docker logs 容器名` | 查看日志 | 查看程序的输出 |
| `docker stop 容器名` | 停止容器 | 关闭程序 |
| `docker restart 容器名` | 重启容器 | 重新启动程序 |
| `docker compose up -d` | 启动所有服务 | 一键启动整套系统 |
| `docker compose down` | 停止所有服务 | 一键关闭整套系统 |
| `docker compose build` | 构建镜像 | 编译安装包 |
| `docker exec -it 容器名 /bin/sh` | 进入容器内部 | SSH 登录到虚拟机 |
