# 微信商户号配置部署操作记录

> 本文档记录了将微信支付商户号配置部署到生产服务器的完整操作步骤，帮助理解前后端支付配置的关联。

---

## 一、背景

拿到微信支付商户号后，需要在服务器上配置商户号、API 密钥和证书，后端才能正常调用微信支付接口。

**前置条件（已完成）：**
- 微信支付商户号已申请通过
- API 密钥（APIv2 和 APIv3）已在 pay.weixin.qq.com 设置
- API 证书已从微信支付商户平台下载
- 商户号已关联到小程序（mp.weixin.qq.com）

---

## 二、本地文件结构

从微信商户平台下载的证书文件（3个）：

```
1103672030_20260601_cert/
├── apiclient_cert.p12      # PKCS12 格式证书（含公钥和私钥）
├── apiclient_cert.pem      # PEM 格式公钥证书
├── apiclient_key.pem       # PEM 格式私钥
└── 证书使用说明.txt
```

> **说明：** APIv3 只需要 `apiclient_key.pem`（私钥）和 `apiclient_cert.pem`（公钥证书），`.p12` 是兼容旧版 APIv2 用的。

## 三、服务器配置步骤

### 1. 连接服务器并将证书上传

```bash
# 创建证书目录
ssh root@guaibaobao.cn "mkdir -p /opt/gbb/certs"

# 上传证书文件（在本地执行）
scp apiclient_cert.pem root@guaibaobao.cn:/opt/gbb/certs/
scp apiclient_key.pem root@guaibaobao.cn:/opt/gbb/certs/
scp apiclient_cert.p12 root@guaibaobao.cn:/opt/gbb/certs/
```

### 2. 修改 docker-compose.yml

在 `backend` 服务中添加卷挂载，将宿主机的证书目录映射到容器内。

```yaml
# /opt/gbb/docker-compose.yml
backend:
  # ... 其他配置 ...
  env_file: /opt/gbb/.env
  volumes:
    - /opt/gbb/certs:/app/certs:ro    # 新增：证书只读挂载
  read_only: true
  # ...
```

> **为什么需要这一步？** 容器设置了 `read_only: true`，文件系统不可写入，无法通过 `docker cp` 复制文件进去。必须通过 volume 挂载方式让容器访问宿主机的证书文件。

### 3. 更新 .env 配置文件

配置在 `/opt/gbb/.env` 中，这些变量通过 `env_file` 注入到容器。

```bash
# 修改前（占位符）
WECHAT_MCH_ID=1234567890
WECHAT_API_KEY=your-wechat-api-key
WECHAT_API_V3_KEY=mock_api_v3_key_for_development_32
WECHAT_SERIAL_NO=mock_serial_number
PAYMENT_MOCK_MODE=true
PAYMENT_SANDBOX_MODE=true

# 修改后（真实值）
WECHAT_MCH_ID=<实际商户号>
WECHAT_API_KEY=<实际 APIv2 密钥>
WECHAT_API_V3_KEY=<实际 APIv3 密钥>
WECHAT_SERIAL_NO=<实际证书序列号>
PAYMENT_MOCK_MODE=false
PAYMENT_SANDBOX_MODE=false
```

执行命令：

```bash
ssh root@guaibaobao.cn 'sed -i "s/WECHAT_MCH_ID=1234567890/WECHAT_MCH_ID=1103672030/" /opt/gbb/.env'
ssh root@guaibaobao.cn 'sed -i "s/PAYMENT_MOCK_MODE=true/PAYMENT_MOCK_MODE=false/" /opt/gbb/.env'
# ... 其他配置项同理
```

### 4. 重建容器使配置生效

由于环境变量是在容器创建时从 `.env` 读取的，只 `restart` 不会生效，必须重建容器：

```bash
# 备份
cp /opt/gbb/docker-compose.yml /opt/gbb/docker-compose.yml.bak.20260601

# 删除旧容器（否则会冲突报错 "container name already in use"）
docker rm -f gbb-backend

# 重新创建容器
docker compose up -d backend
```

> **常见错误：** `docker compose up -d --force-recreate backend` 会报容器名冲突，因为旧容器还在。必须先 `docker rm -f` 删除旧容器再重建。

### 5. 重新加载 nginx

容器重建后 IP 地址会变，而 nginx 在启动时解析 `gbb-backend` 主机名并缓存了旧 IP。需要重新加载 nginx：

```bash
docker exec gbb-nginx nginx -s reload
```

> **如果不执行这一步：** 会看到 `502 Bad Gateway`，nginx 错误日志中显示 `connect() failed (113: Host is unreachable)`。

### 6. 验证

```bash
# 检查环境变量是否注入
docker exec gbb-backend env | grep -E 'WECHAT_|PAYMENT_'

# 检查证书文件是否可访问
docker exec gbb-backend ls -la /app/certs/

# 测试 API 健康检查
curl https://guaibaobao.cn/api/v1/health
# 返回: {"status":"ok","services":{"database":"connected","redis":"connected"}}
```

## 四、涉及的关键文件

| 文件 | 位置 | 用途 |
|------|------|------|
| docker-compose.yml | `/opt/gbb/docker-compose.yml` | 容器编排，定义 volume 挂载和环境变量来源 |
| .env | `/opt/gbb/.env` | 存放敏感配置（数据库密码、微信密钥等） |
| 证书文件 | `/opt/gbb/certs/apiclient_*.pem` | 微信支付 API 证书，通过 volume 挂载进容器 |
| nginx 配置 | `/opt/gbb/nginx/nginx.conf` | 反向代理，`proxy_pass` 到 `gbb-backend:3000` |

## 五、注意事项

### 容器 read_only 模式

生产环境容器通常开启 `read_only: true` 增加安全性。这意味着：

- 容器内不能写文件（除 tmpfs 挂载点外）
- **不能**用 `docker cp` 或 `docker exec` 写入证书文件
- **必须**用 volume 挂载让容器访问外部文件

### nginx DNS 缓存

nginx 在启动时通过 DNS 解析 upstream 主机名（`gbb-backend`），并缓存结果。容器重建后 IP 改变，需要：

1. `docker exec gbb-nginx nginx -s reload` 刷新
2. 或者配置 nginx 使用变量方式动态解析（更复杂，一般不必要）

### 环境变量变更需要重建容器

- `docker restart` → 只重启进程，**不会**重新读取 `.env` 或 `docker-compose.yml` 中的 `environment`
- `docker compose up -d` → 重新读取配置，**会**应用新的环境变量

  3. 退款等操作必须用
  查询余额、退款、转账这些涉及资金的操作，微信要求必须使用这个证书（不能用密钥签名代替）。

  证书像身份证+公章。
