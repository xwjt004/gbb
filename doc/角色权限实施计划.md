# 角色权限系统实施计划

## 现状概览

权限系统的基础设施（`PermissionGuard`、`RequirePermission` 装饰器、`Role`/`RolePermission` 模型、角色 CRUD 接口）**已存在但未真正启用**。当前所有用户可访问全部功能，无实际权限控制。

### 现存问题

| 问题 | 位置 | 影响 |
|------|------|------|
| `PermissionGuard` 未注册为全局守卫 | `src/shared/guards/` | 所有接口无权限拦截 |
| `@RequirePermission()` 未在任何控制器使用 | 全部 controller | 同上 |
| JWT payload 不含角色/权限 | `auth.service.ts` / `users.service.ts` | 前端无法从 token 判断权限 |
| `UpdateUserDto` 缺少 `roleId` | `users/dto/update-user.dto.ts` | 前端分配角色不生效 |
| 前端权限常量和后端格式不一致 | `constants.ts` vs `roles.service.ts` | `user:view` vs `users:view` |
| `RequirePermission` 前端组件未使用 | `components/ProtectedRoute/` | 路由无权限拦截 |
| 侧边栏菜单未做权限过滤 | `components/Layout/` | 所有菜单对所有用户可见 |
| 权限树不完整 | `roles.service.ts` | 缺少 CRM、采购、库存等模块 |
| 默认角色不完整 | `roles.service.ts` | 缺少店长、摄影师、后期 |

---

## 第一阶段：基础设施修复

> 目标：让已有的权限体系真正运转起来。涉及后端 + 前端同步改造。

### 1.1 统一权限格式

**后端权限树**（`roles.service.ts` `getPermissionTree()`）使用复数 module 格式如 `users:view`，**前端常量**（`constants.ts` `PERMISSIONS`）使用单数格式如 `user:view`。需统一。

**操作**：
- 以前端 `constants.ts` 中的 `PERMISSIONS` 常量为蓝本，扩展为完整权限列表
- 格式统一为 `module:action`（与后端一致）
- 后端 `getPermissionTree()` 补充缺失模块并与之对齐

### 1.2 修复 JWT 与登录链路

**操作**：
- 修改 `users.service.ts` `adminLogin`：登录时查询用户角色和权限列表，写入 JWT payload
- JWT payload 新增字段：`roleId`, `roleName`, `permissions: string[]`
- 修改 `authSlice.ts`：扩展 `User` 接口，加入 `roleId`, `roleName`, `permissions`
- 修改 `ProtectedRoute`：从 store 中读取权限并传递给子组件

### 1.3 启用后端权限守卫

**操作**：
1. 在 `AppModule` 中将 `PermissionGuard` 注册为全局 `APP_GUARD`
2. 无 `@RequirePermission()` 装饰器的接口默认放行（向后兼容）
3. 按优先级给关键接口添加 `@RequirePermission()` 装饰器

**首批需要加权限的接口**：

| 模块 | 接口 | 权限 |
|------|------|------|
| 订单 | `POST /orders`, `PATCH /orders/:id` | `orders:edit` |
| 订单 | `DELETE /orders/:id` | `orders:delete` |
| 订单 | `POST /orders/:id/approve` | `orders:approve` |
| 订单 | `POST /orders/:id/refund` | `orders:refund` |
| 用户 | `POST /users`, `PATCH /users/:id` | `users:edit` |
| 用户 | `DELETE /users/:id` | `users:delete` |
| 套系 | `POST /packages`, `PATCH /packages/:id` | `packages:edit` |
| 套系 | `DELETE /packages/:id` | `packages:delete` |
| 支付 | `POST /payments/refund` | `payments:refund` |
| 角色 | `POST /roles`, `PATCH /roles/:id` | `system:roles` |
| 角色 | `DELETE /roles/:id` | `system:roles` |

### 1.4 修复角色分配链路

**操作**：
- `UpdateUserDto` 增加 `roleId?: number` 字段
- `users.service.ts` update 方法处理 `roleId` 关联更新
- 验证：前端 UserForm 选择角色 → 提交 → 数据库成功写入

### 1.5 启用前端路由/菜单权限

**操作**：
1. 修改 `Layout/index.tsx` 侧边栏菜单：根据 `user.permissions` 过滤菜单项
2. 修改 `App.tsx` 路由配置：对敏感路由使用 `RequirePermission` 包裹
3. 菜单项和路由共用同一个权限 Map

## 第三阶段：前端权限交互

### 3.1 角色管理页面

**现有页面**（`src/pages/System/Roles/`）已有角色 CRUD，需要增强：

- 权限选择器改用新权限树（按模块分组展示，树形 checkbox）
- 显示每个角色的关联用户数（已有）
- 增加角色状态（启用/停用）切换
- 系统角色禁用删除按钮

### 3.2 用户管理页面权限化

**现有页面**（`src/pages/Users/`）：

- UserList：根据权限显示/隐藏操作按钮（新增、编辑、删除、分配角色）
- UserForm：角色选择下拉加载角色列表（已有，需要验证 `roleId` 提交链路）
- 用户详情页显示当前角色和权限

### 3.3 页面级权限提示

- 无权限访问某路由时，显示 403 无权限页面（而不是空白或跳转）
- 无权限执行某操作时，按钮 disabled + tooltip 提示"请联系管理员开通权限"

### 3.4 前端权限常量完善

更新 `constants.ts` `MENU_ITEMS` 和 `PERMISSIONS`：

- `MENU_ITEMS` 当前仅定义了 9 个条目，实际 Layout 有 20+ 菜单项
- 所有菜单项补全 `permission` 字段
- 侧边栏根据 `user.permissions` 动态过滤

## 第四阶段：深化与优化

### 4.1 操作日志

- 角色变更（创建/修改/删除）写入操作日志
- 操作日志记录：操作人、操作类型、变更详情、时间
- 使用现有的 `OperationLogInterceptor`

### 4.2 数据级权限

- 摄影师：订单列表只显示自己参与的拍摄订单
- 后期：订单列表只显示待取件的订单
- 实现方式：在 service 层根据 `req.user.roleId` 追加查询条件

### 4.3 审批流（可选）

- 采购单需要店长审批才能进入下一步
- 退款超过一定金额需要店长审批
- 结合状态机（`auto-status-transition`）实现

### 4.4 权限审计

- 定期报告：每个角色下有多少用户、权限覆盖率
- 权限变更通知：角色修改时通知相关用户

## 风险与注意事项

1. **向后兼容**：第一阶段必须保证所有现有接口在无 `@RequirePermission()` 装饰器时行为不变
2. **数据迁移**：已有用户没有角色，需设置兜底行为（默认分配「管理员」角色或明确无权限）
3. **测试策略**：每个权限变更需要测试正向（有权限能操作）和负向（无权限被拒绝）两种情况
4. **前端缓存**：权限变更后用户需要重新登录才能获取新的 JWT
